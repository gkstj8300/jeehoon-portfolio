## 바이브 코딩이라는 단어를 처음 들었을 때

OpenAI 공동창업자 Andrej Karpathy가 2025년 초에 트위터에 올린 글이 개발자 커뮤니티에서 화제가 되었어요. 코드를 직접 타이핑하는 대신 AI에게 자연어로 원하는 것을 설명하고 AI가 생성한 코드를 수용하며 개발하는 방식. 그는 이것을 `바이브 코딩(Vibe Coding)`이라고 불렀죠.

```point
- 바이브 코딩(Vibe Coding)이란?
AI에게 자연어로 원하는 기능을 설명하고 AI가 생성한 코드를 검토하며 개발을 진행하는 방식이에요.
```

처음 이 개념을 접했을 때 두 가지 감정이 동시에 들었어요. "이거 정말 편하겠다"라는 기대감, 그리고 "그러면 개발자의 역할은 뭐가 되는 거지?"라는 불안감이었죠.

## 바이브 코딩은 어떻게 작동하는 걸까

기존 개발 방식은 항상 이런 과정을 거치죠.

```list
1. 요구사항을 분석한다
2. 설계를 고민한다
3. 코드를 직접 작성한다
4. 에러를 디버깅한다
5. 리팩토링한다
```

바이브 코딩에서는 이 과정이 달라져요. 개발자가 "무엇을 만들고 싶은지"를 자연어로 설명하면 AI가 코드를 생성하고 개발자는 그 결과를 확인하며 방향을 잡아가죠. Karpathy는 "코드를 읽지 않고 느낌(vibe)으로 간다"고 표현했어요.

현재 이런 도구들이 이미 개발 환경에 자리를 잡아가고 있어요.

```list
- GitHub Copilot: 코드 자동완성과 제안
- Cursor: AI 기반 코드 에디터
- Claude Code: 터미널에서 자연어로 코딩하는 CLI 도구
- ChatGPT / Claude: 코드 생성 및 리뷰
```

저도 실무와 개인 프로젝트에서 AI 도구를 활용하고 있어요. 반복적인 보일러플레이트 코드를 작성하거나 익숙하지 않은 라이브러리를 빠르게 파악할 때 생산성 향상을 크게 느끼고 있죠.

## 직접 경험하며 느낀 것들

개인 블로그 프로젝트에서 AI 도구를 활용한 경험이 있어요. 마크다운 파일에서 프론트매터(Front Matter)를 파싱하고 정렬하는 로직을 만들 때가 좋은 예시죠. 이전이라면 gray-matter 라이브러리 문서를 읽고 예제를 찾아보고 직접 코드를 작성했을 거예요. 바이브 코딩 방식으로는 "gray-matter로 마크다운 파일들을 읽어서 날짜순으로 정렬하고 PostType 인터페이스에 맞게 매핑해줘"라고 설명하면 거의 완성된 코드가 나와요.

```caution
바이브 코딩이 효과적인 경우와 주의해야 할 경우를 구분하는 것이 중요해요.
```

다만 AI가 생성한 코드를 무조건 수용하면 안 돼요. AI가 생성한 코드가 동작은 하지만 최적의 방법이 아닌 경우가 자주 있었어요. 타입 정의가 느슨하거나 불필요한 상태 관리를 추가하거나 프로젝트의 기존 패턴과 맞지 않는 구조를 제안하기도 했죠.

그래서 저는 바이브 코딩을 할 때도 생성된 코드를 읽고 왜 이렇게 작성되었는지를 이해하려고 해요. 이 과정이 없으면 내 프로젝트 안에 내가 이해하지 못하는 코드가 쌓이니까요.

## 바이브 코딩이 잘 맞는 영역

모든 개발 작업에 바이브 코딩이 적합한 것은 아니에요. 특히 효과적인 영역이 있죠.

```list
- 보일러플레이트 코드: 컴포넌트 초기 구조, CRUD API 라우트, 타입 정의 등
- 스타일링: CSS/SCSS 레이아웃 작업, 반응형 미디어 쿼리 작성
- 테스트 코드: 기존 로직에 대한 유닛 테스트 생성
- 리팩토링: 기존 코드의 구조 개선이나 패턴 변환
- 문서화: JSDoc, README 등 문서 작성
```

반면 복잡한 비즈니스 로직, 성능에 민감한 최적화, 보안이 중요한 인증/인가 로직에서는 AI가 생성한 코드를 더 꼼꼼히 검토해야 해요. AI는 프로젝트의 전체 맥락을 파악하지 못하기 때문에 개발자가 직접 판단해야 하죠.

## 그렇다면 개발자의 역할은 어떻게 바뀌는 걸까

퇴사 후 회고록에서 "개발자들의 능력은 어디서 나뉘어질까?"라는 질문을 한 적이 있어요. AI가 코드를 대신 작성해주는 시대에 이 질문은 더 절실해졌죠.

바이브 코딩 시대에 개발자에게 더 중요해지는 능력이 있어요.

```list
1. 문제 정의 능력: AI에게 무엇을 만들어야 하는지 명확하게 설명하는 것
2. 아키텍처 설계: 전체 구조를 그리고 각 모듈의 역할을 정하는 것
3. 코드 리뷰 역량: AI가 생성한 코드의 품질을 판단하는 것
4. 디버깅 능력: 문제가 발생했을 때 원인을 파악하고 해결하는 것
5. CS 기초 지식: 왜 이 방식이 더 나은지를 판단할 수 있는 근거
```

결국 깊이감이에요. AI가 아무리 코드를 잘 생성해도 "이 코드가 왜 이렇게 동작하는지", "이 구조가 정말 최선인지", "성능상 병목은 없는지"를 판단하는 것은 개발자의 몫이니까요.

저는 공통 컴포넌트를 설계할 때도 AI에게 초안을 요청해요. 하지만 props 인터페이스를 어떻게 가져갈지, 확장성과 제한 사이의 균형을 어디에 둘지는 직접 고민하고 결정하죠. 이런 설계 판단은 아직 사람의 영역이에요.

## AI 도구를 잘 활용하기 위한 팁

바이브 코딩을 활용하며 실천하고 있는 방법 세 가지를 공유할게요.

첫 번째, 프롬프트를 구체적으로 작성해요. "버튼 컴포넌트 만들어줘"보다는 "React, TypeScript 환경에서 theme, size, icon props를 받는 공통 Button 컴포넌트를 forwardRef로 감싸서 만들어줘. 스타일은 SCSS Modules를 사용해"처럼요. 구체적으로 작성할수록 원하는 결과에 가까운 코드가 나와요.

두 번째, 생성된 코드를 반드시 읽어요. 바이브 코딩의 핵심이 "코드를 읽지 않는 것"이라고 하지만 실무에서는 다르죠. 핵심 로직은 이해하고 넘어가야 문제가 생겼을 때 대응할 수 있어요.

세 번째, AI를 페어 프로그래밍 파트너로 활용해요. 혼자 고민하며 막혀있을 때 AI에게 "이 구조에 대해 어떻게 생각해?"라고 물어보면 새로운 관점을 얻을 수 있죠. 다만 최종 결정은 항상 개발자가 해야 해요.

## 마지막으로

바이브 코딩은 개발의 진입 장벽을 낮추고 생산성을 높여주는 강력한 도구예요. 하지만 도구일 뿐이에요. 망치가 아무리 좋아도 어디에 못을 박을지는 사람이 결정하듯이 AI가 코드를 잘 생성해도 무엇을, 왜 만들어야 하는지를 결정하는 것은 개발자의 역할이에요.

AI 시대에도 변하지 않는 것이 있어요. 기술의 원리를 이해하려는 태도, 더 나은 구조를 고민하는 습관, 끊임없이 배우려는 자세. 바이브 코딩이라는 새로운 물결 속에서도 이 기본기가 개발자의 경쟁력을 결정짓죠.

AI와 함께 코드를 만들되 코드의 주인은 항상 개발자여야 해요. 이것이 제가 바이브 코딩을 경험하며 내린 결론이에요.
